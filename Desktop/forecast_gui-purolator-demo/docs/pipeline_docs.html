01The full pipeline — end to end

This diagram shows every step from raw data to finalized daily forecast. The Volume Planner sits at step 4 — the human review step. Everything before it is Mehrdad's existing R/Python scripts. Everything after is what the app produces.

01
Courier Ops
QlikView pull every Monday. Daily piece counts per terminal. Jan 01 → end of last week.
02
FMR Daily DB
Pre-processing script. Historical (2012-2025) + Current year combined. 135 terminals × 13 years × 365 days.
03
FMR Weekly + Normalized
Aggregate daily → weekly. Normalize. Feed to R model.
04
Volume Planner
Review chart. Adjust weeks. Validate daily shapes. Export finalized forecast.
05
Daily Distribution
Multiply normalized weekly forecast by daily % allocation per terminal.
06
Final Daily Forecast
Finalized daily piece counts per terminal. Goes to ops planning.
What the app does not run: Steps 1-3 are Mehrdad's external scripts. The app receives the output of step 3 (Actual.xlsx and Forecast.xlsx) as uploads. Steps 1-3 can optionally be triggered from within the app once the FMR pre-processing script is wired in.

02FMR pre-processing — what the app now does

Previously Mehrdad ran this step externally. The app now accepts both raw input files and runs the combine+aggregate step internally. The logic exactly mirrors what Mehrdad described on the whiteboard.

# Pseudocode for what /api/upload_fmr_historical and /api/upload_fmr_current do

# Step 1: Load both files
hist_df  = pd.read_parquet("historical_fmr_2012_2025.parquet")   # ~640k rows
curr_df  = pd.read_csv("current_year_2026_w01_w24.csv")             # ~1344 rows

# Step 2: Stack them (current year may have updated rows for tail of historical)
daily    = pd.concat([hist_df, curr_df])
daily    = daily.drop_duplicates(
              subset=["Year", "Week.Number", "Day", "Terminal"],
              keep="last"   # current year wins over historical
           )

# Step 3: Aggregate daily → weekly (sum pieces per terminal per week)
METRICS  = ["PCL.Del.Pcs.N", "Agent.Del.Pcs.N", "Total.Del.Pcs.N"]
weekly   = (daily
             .groupby(["Year", "Week.Number", "Terminal", "Division"])
             [METRICS].sum()
             .reset_index()
          )

# This weekly frame is now equivalent to Actual.xlsx
# It is stored in memory and used by the chart and table endpoints
store["actual"] = weekly
Result: After uploading both files, the app automatically runs this and populates the weekly actuals used by the chart. The metric selector, division filter, and terminal list all populate from this combined data. No manual export/import needed.
03The R model — what it does and how the app replicates it

Mehrdad's original model is an STL+ARIMA with external regressors, fit per terminal per metric in R. The app includes a Prophet-based equivalent for the 2027 forecast overlay. This section explains both.

Mehrdad's R model (STL + ARIMA + xreg)

# R pseudocode — Mehrdad's actual model structure
# Fit per terminal, per metric. Then aggregate.

for terminal in terminals:
  for metric in [PCL.Del.Pcs.N, Agent.Del.Pcs.N]:

    # 1. Build training series from normalized FMR weekly
    y       = normalized_fmr_weekly[terminal, metric, year >= 2018]
    xreg    = cbind(cyber, catchup, amazon)   # external regressors

    # 2. Decompose with STL (Seasonal-Trend decomposition)
    stl_fit = stl(y, s.window="periodic")
    remainder = stl_fit$time.series[, "remainder"]

    # 3. Fit ARIMA on the remainder with external regressors
    arima_fit = auto.arima(remainder, xreg=xreg, stepwise=FALSE, approximation=FALSE)

    # 4. Forecast 52 weeks into 2027
    xreg_future  = cbind(cyber_2027, catchup_2027, amazon_2027)
    fc           = forecast(arima_fit, h=52, xreg=xreg_future)

    # 5. Re-add STL seasonal component to ARIMA forecast
    seasonal_component = stl_fit$time.series[, "seasonal"]
    final_forecast = fc$mean + seasonal_component

    # 6. Clip negatives (convert_to_positive)
    final_forecast = pmax(final_forecast, 0)

    results[terminal, metric] = final_forecast

# Total = PCL + Agent (never fit directly)
results[terminal, Total] = results[terminal, PCL] + results[terminal, Agent]

The app's Prophet equivalent

The app includes a Python Prophet model that produces a 2027 forecast overlay. It doesn't replace the R model but provides a visual comparison. The logic is in prophet_pipeline/.

# prophet_pipeline/model.py — simplified
from prophet import Prophet

def fit_and_predict(train, future):
    model = Prophet(
        yearly_seasonality = True,
        weekly_seasonality = False,    # data is already weekly
        seasonality_mode   = "multiplicative",
        interval_width     = 0.8
    )
    # Add same regressors as R model
    model.add_regressor("cyber")      # Cyber Monday week flag
    model.add_regressor("catchup")    # Post-Cyber catch-up week flag
    model.add_regressor("amazon")     # Amazon weekly volume (continuous)

    model.fit(train)
    forecast = model.predict(future)
    result["yhat"] = forecast["yhat"].clip(lower=0)
    return result

R model strengths

  • STL handles the seasonal decomposition explicitly
  • ARIMA captures autocorrelation in the remainder
  • Per-terminal tuning (starting years, ARIMA order)
  • Proven on Purolator data for multiple years
  • 8 variables with different starting years handled

Prophet model (app) strengths

  • Runs inside the app — no R installation needed
  • Same three regressors: Cyber, Catchup, Amazon
  • Multiplicative seasonality handles Purolator's Q4 spikes
  • Cache invalidates on re-upload — always fresh
  • 2027 overlay visible on same chart as 2026 for direct comparison
04Regressors — Cyber Week, Amazon, and how they work

Both the R model and the Prophet model use three external regressors. These are variables known in advance (for 2027) that meaningfully shift volume beyond what the seasonal pattern alone predicts.

RegressorTypeSource fileWhat it captures2027 approach
cyber Binary flag (0/1) CyberWeekRegressor.xlsx Cyber Monday week (W48 historically). Volume spikes 15-25% above seasonal baseline. Without this regressor the model undershoots peak badly. Defaults to W48. Update config.CYBER_WEEK_2027 once Purolator's 2027 fiscal calendar is confirmed.
catchup Binary flag (0/1) CyberWeekRegressor.xlsx The week after Cyber (W49). Some volume delayed from W48 ships in W49. Smaller effect than Cyber but still significant. Defaults to W49. Same update process as cyber.
amazon Continuous (weekly volume) AmazonRegressor.xlsx Amazon's weekly parcel volume flowing through Purolator. Strong Q4 uplift (W47-W51) and Q1 reduction (W1-W4 "return season"). Continuous so the model captures the ramp-up and ramp-down shape, not just a flag. 2027 projected from 2026 using 2025→2026 YoY growth ratio per week. Clamped to 0.5×-1.5× to prevent noise from single-week anomalies.

2027 regressor projection logic

Neither source file has 2027 data yet. The app generates it:

# prophet_pipeline/regressors.py — _build_2027_amazon()

# Method: "per_week" (default) or "overall"
# Per-week: ratio = amazon_2026[week] / amazon_2025[week] → applied to 2026
merged["ratio"] = amazon_2026 / amazon_2025   # YoY growth per week
merged["ratio"] = merged["ratio"].clip(
    lower=0.5,    # never project less than 50% of prior year
    upper=1.5     # never project more than 150% (guards against anomaly weeks)
)
amazon_2027 = amazon_2026 * merged["ratio"]

# Cyber/Catchup for 2027: default W48/W49 (same as 2026)
# Update config.CYBER_WEEK_2027 and CATCHUP_WEEK_2027 when calendar is confirmed
Update required each January: Set config.CYBER_WEEK_2027 and config.CATCHUP_WEEK_2027 to the real fiscal week numbers once Purolator publishes the 2027 fiscal calendar. Also update AMAZON_2027_GROWTH_METHOD if the Amazon growth pattern changes significantly.

How to update: On the Daily Distribution page, scroll to the "Regular Week Assignments" card. Select a division, enter the year and comma-separated week numbers, click Save. Export CSV to keep a record. The following January, update the weeks to reflect the new year's holiday-free weeks.

How these weeks are used

For each terminal, the app (and Mehrdad's script) takes the daily distribution for each of the regular weeks and averages them. The result is the "regular week" template — the Mon-Sun percentage shape applied to any non-event week for that terminal.

# For terminal 101 in GTA (regular weeks = W16, W17, W18, W22 of 2026)
reg_weeks   = [16, 17, 18, 22]
daily_dists = [daily_dist[terminal=101, year=2026, week=w] for w in reg_weeks]

# Average across the 4 weeks
avg_dist = mean(daily_dists)   # shape: {Sun: 1%, Mon: 20%, Tue: 21%, ...}

# Normalize to 100%
reg_template = avg_dist / avg_dist.sum() * 100
07Daily distribution workflow — regular vs event weeks

From the whiteboard: daily distribution splits into two branches. The app handles both.

Regular weeks (Non-Holiday)

All 52 weeks except the event weeks. One standard Mon-Sun shape per terminal per year. Derived from the regular week assignments (Section 06). Does not change during the year unless a planner manually overrides it.

  • Typical shape: Sun ~1%, Mon-Thu ~19-22% each, Fri ~17%, Sat ~2.5%
  • Sums to exactly 100%
  • Set once per year per terminal

Event weeks (Holiday + Peak-Prime)

Canada Day (W27), Labour Day (W36), Thanksgiving (W41), Remembrance Day (W45), Peak/Prime (W47-49), Christmas (W51-52). Each has a different shape because holidays shift which days carry volume.

  • Canada Day (Monday holiday): Mon near 0%, Tue-Fri elevated
  • Peak W47-49: Mon-Thu all elevated, Sat also higher than usual
  • Derived from 2-3 historical base years with same holiday weekday
  • Aggregated and normalized to 100%

The overlay system — how it works on screen

  1. Navigate to event week (e.g. W27 Canada Day) on the Daily Distribution page
  2. App auto-loads base weeks from event map: 2015 W27, 2020 W28
  3. Blue solid line = current week distribution
  4. Purple dashed = 2015 W27, Green dashed = 2020 W28
  5. Orange bold = aggregated average of all overlays, normalized to 100%
  6. Visually confirm the orange line looks right
  7. If not — manually adjust in the edit table below
  8. Save → downloads updated CSV → re-upload next session
The key constraint Mehrdad stated: Summation MUST be 100% per terminal per week per metric. The app enforces this: the Total row turns red if off, the Normalize button proportionally adjusts all days, and Save refuses to proceed if the sum is not within 0.05% of 100%.
